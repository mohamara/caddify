# Contributing to caddify

Thanks for helping improve caddify. Small, focused changes are easiest to
review and ship.

## Code of conduct

By participating, you agree to follow the [Code of Conduct](CODE_OF_CONDUCT.md).

## How to report bugs

Open an issue with:

1. What you expected vs what happened
2. OS and Docker version (`docker version`, `docker compose version`)
3. Relevant commands (`./proxy status`, `./proxy logs`)
4. Sanitized snippets of `routes.conf` / Caddyfile (no secrets)

For security issues, see [SECURITY.md](SECURITY.md) — do not file public issues.

## How to propose features

Open an issue first for non-trivial ideas. Describe the problem, not only the
solution. Keep scope small: this project aims to stay a thin layer over Caddy +
Docker.

## Development setup

```bash
git clone https://github.com/mohamara/caddify.git
cd caddify
cp .env.example .env
# edit .env — especially DASHBOARD_PASSWORD and ACME_EMAIL
./proxy setup you@example.com
```

Useful commands while developing:

```bash
./proxy status
./proxy logs
./proxy apply          # regenerate Caddyfile + reload
docker compose ps
```

Dashboard source lives in `dashboard/` (FastAPI). After UI or API changes:

```bash
docker compose up -d --build dashboard
```

CLI logic lives in the `./proxy` bash script. Prefer matching behavior between
CLI and dashboard when you change route format or SSL flags.

## Pull requests

1. Branch from the default branch
2. Keep the PR focused (one concern per PR)
3. Update the README if you change user-facing behavior
4. Test locally:
   - `./proxy add` / `set` / `rm` / `list`
   - SSL on and `--no-ssl`
   - Dashboard add / update / delete / reload
5. Use conventional commit messages when possible:
   - `feat: …`
   - `fix: …`
   - `docs: …`
   - `chore: …`
   - `refactor: …`
   - `test: …`

### What we look for in reviews

- Correctness and clear failure messages
- No secrets in the diff
- Compatibility with Docker Compose v2
- Docs that match the code

## Project layout

```
.
├── proxy                 # CLI entrypoint
├── bin/                  # thin wrappers (up, apply, status)
├── routes.conf           # domain → port map (generated / edited)
├── caddy/Caddyfile       # generated by CLI or dashboard
├── docker-compose.yml    # Caddy + dashboard
├── dashboard/            # FastAPI web UI
└── .env.example          # documented environment variables
```
